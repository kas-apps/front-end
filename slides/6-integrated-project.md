# バイブコーダーに贈るフロントエンド開発入門

## Phase 6: 統合プロジェクト 🚀

---

## Phase 6 の目標

**AIと協働**しながら、ゼロから本格的なWebアプリケーションを作り上げる

### Phase 6 でできるようになること

- AIと対話しながら、作りたいものの要件を明確化できる
- 要件を基に、技術仕様書やデータ構造を設計できる
- AIに効果的な指示を出して、機能を段階的に実装できる
- 中規模アプリケーション（1000-1500行）を完成させられる
- 学んだパターンを応用して、自分のオリジナルプロジェクトを始められる
- Phase 1~5の知識を統合的に活用できる

###### アイデアを形にし、実用的なWebアプリケーションを自分で作れるようになる！

---

## Phase 6 の特徴 🎯

### 従来の教材との違い

従来のプログラミング教材では「完成品を最初に見せて、それを作る方法を教える」のが一般的
この教材では、**AIと一緒に「ゼロから作り上げる」プロセスそのものを学ぶ**のが、Phase 6の最大の特徴！

### Phase 6 で体験すること

1. **要件定義**：AIと対話しながら「**何を作るか**」を明確にする
2. **仕様設計**：データ構造やUI設計をAIと一緒に考え「**どう作るか**」を明確にする
3. **段階的実装**：基本機能から始めて、少しずつ機能を追加していく
4. **改善・最適化**：動くコードをさらに良いコードにブラッシュアップする

この手順は、**実際のWeb開発の現場で行われているプロセスそのもの**だよ！

# Lesson 1

# プロジェクトの要件定義 📋

---

## Lesson 1 : プロジェクトの要件定義 📋

**学習目標**：AIと対話しながらプロダクトのアイデアを具体化し、要件を定義できるようになる

### なぜ要件定義が重要か

プログラミングを学び始めた人が陥りがちな罠、それは **「いきなりコードを書き始めてしまう」** こと！実は、**良いプロダクトを作るには、コードを書く前の準備が超重要**なんだ！

### 要件定義とは？

要件定義とは、**「何を作るか」を明確にすること**だよ。

- どんな機能が必要？
- 誰が使う？
- どんな問題を解決する？
- 必須の機能は？あったらより良い機能は？

これらを最初に整理しておくことで、**迷わず開発を進められる**んだ！

---

## AI時代の要件定義

従来は、要件定義には専門知識が必要だったけど、**AI時代は違う！**

AIと対話しながら、以下のようなことができるよ：

- 「こんなアプリを作りたい」というアイデアをAIに相談
- AIから機能の提案をもらう
- 実装の難易度をAIに聞く
- 優先順位をAIと一緒に考える

**AIは最高のブレインストーミングパートナー！** 🤖

---

## 要件定義のステップ

### ステップ1：アイデアの明確化

作りたいものを大まかに決める：

- 何のアプリ？（TODO、レシピ、家計簿など）
- どんな人が使う？（自分、友達、不特定多数など）
- どんな問題を解決する？（タスク管理、料理の記録など）

### ステップ2：AIとのブレインストーミング

AIに相談して、機能のアイデアを広げる：

- どんな機能があると便利？
- 似たアプリにはどんな機能がある？
- 実装が簡単な機能は？難しい機能は？

---

### ステップ3：ユーザーストーリーの作成

「〜として、〜したい、なぜなら〜」という形式でストーリーを書く

```text
例：ユーザーとして、タスクに優先度を設定したい、なぜなら重要なタスクを見逃さないようにするため
```

### ステップ4：機能の優先順位付け

機能を以下のように分類する：

- **必須機能（Must Have）**：これがないとアプリとして成り立たない
- **拡張機能（Should Have）**：あると便利だけど、なくても動く
- **オプション機能（Nice to Have）**：余裕があれば実装したい

### ステップ5：要件定義書の作成

整理した内容を文書化する（マークダウン形式推奨）

---

## 事例：カード型TODOアプリの要件定義プロセス

実際に、カード型TODOアプリの要件定義を進めてみよう！

### ステップ1：アイデアの明確化

```text
何を作る？
→ TODOアプリ（タスク管理アプリ）

誰が使う？
→ 自分自身（個人のタスク管理）

どんな問題を解決する？
→ やるべきことを忘れないようにする
→ タスクの優先度を可視化する
→ 完了したタスクを記録する

シンプルなリスト形式のTODOリストとの違いは？
→ カード形式で視覚的に見やすく
→ より多くの情報（説明、優先度、カテゴリ、締切日）
→ フィルターやソートなどの高度な機能
```

---

### ステップ2：AIとのブレインストーミング

**🤖 AIとの協働（最重要セクション）**

ここで、AIと対話して機能のアイデアを広げるよ！

#### AIへの指示例

```text
「中規模のTODOアプリを作りたいです。

【背景】
- シンプルなリスト形式のTODOアプリよりも高機能で、実用的なものを作りたいです
- カード形式で視覚的に見やすくしたいです

【質問】
どんな機能があると実用的なTODOアプリになりますか？
必須機能と拡張機能に分けて提案してください。
また、それぞれの実装難易度も教えてください。」
```

---

#### AIとの対話の進め方

要件定義フェーズでは、以下のようにAIと対話を重ねていくよ：

- 機能の提案をもらう
- 機能の詳細を掘り下げる
- 優先順位を相談する

#### 生成された提案のチェックポイント

AIから機能提案をもらったら、以下をチェックしよう：

- **実装可能性**
- **必須 vs 拡張の分類**
- **優先順位**
- **具体性**

# Lesson 2

# 仕様書の作成 📐

---

## Lesson 2: 仕様書の作成 📐

**学習目標**：AIと対話しながら要件を技術仕様に落とし込み、画面構成を考え、データ構造を設計できるようになる

### なぜ仕様作成が重要か

要件定義で「**何を作るか**」が明確になったね！でも、「**作りたいもの**」と「**実際のコード**」の間には、まだ大きなギャップがあるんだ！
**仕様書は、要件と実装をつなぐ橋渡し**なんだ！

```text
✅ 仕様書があれば...

「仕様書を見る」
↓
優先度：'high'/'medium'/'low' の文字列
ID：タイムスタンプ（Date.now()）
日付：ISO 8601形式（'2025-12-01T10:00:00.000Z'）
↓
迷わず実装できる！✨
```

**仕様書があれば、AIへの指示も超具体的になる**んだ！

---

## 仕様書作成のステップ

### ステップ1：データ構造の設計

- タスクオブジェクトのプロパティを決める
- 各プロパティのデータ型を決める
- デフォルト値を決める

### ステップ2：画面構成の設計

- どんな画面要素が必要か整理する
- 各要素のID/クラス名を決める
- レイアウトの方針を決める

---

### ステップ3：LocalStorageの仕様決定

- どのキー名でデータを保存するか
- どのタイミングで保存・読み込みするか
- エラー時の処理を決める

### ステップ4：技術仕様書にまとめる

- 設計した内容をマークダウンで文書化
- コード例を含める
- AIに相談しながら最適化

---

## 事例：カード型TODOアプリの仕様書作成プロセス

### ステップ1：データ構造の設計

**🤖 AIとの協働（最重要セクション）**

#### AIへの指示例

仕様書作成フェーズでは、AIに「技術的な相談」をすることが多いよ。

```text
「以下の要件を実現するために、データ構造を設計したいです。

【要件】
- タスクにはタイトル、説明、優先度、カテゴリ、締切日がある
- タスクは完了/未完了を切り替えられる
- 優先度は「高・中・低」の3段階
- カテゴリは「仕事」「プライベート」「買い物」の3種類
- 締切日は任意（設定しない場合もある）

【制約】
- JavaScriptのオブジェクトと配列で表現する
- LocalStorageに保存できる形式にする
- 一意のIDが必要

JavaScriptのオブジェクト構造を、具体例と一緒に提案してください。
各プロパティのデータ型と、その選択理由も教えてください。」
```

---

#### タスクオブジェクトの設計

```text
要件定義より：
- タスクタイトル（必須）
- タスクの説明（任意）
- 優先度（高・中・低）
- カテゴリ（仕事・プライベート・買い物）
- 締切日（任意）
- 完了フラグ（完了/未完了）
```

これをJavaScriptのオブジェクトに落とし込む：

```javascript
const task = {
  id: number,                    // 一意のID（タイムスタンプ）
  title: string,                 // タスクタイトル
  description: string,           // タスクの説明
  priority: 'high' | 'medium' | 'low',  // 優先度
  category: 'work' | 'personal' | 'shopping',  // カテゴリ
  dueDate: string | null,        // 締切日（YYYY-MM-DD形式、または null）
  completed: boolean,            // 完了フラグ
  createdAt: string              // 作成日時（ISO 8601形式）
};
```

---

#### データ型の選択理由

| プロパティ | データ型 | 理由 |
|-----------|---------|------|
| id | number | Date.now()でタイムスタンプを使用。一意性が保証される |
| title | string | テキスト情報 |
| description | string | テキスト情報（空文字列可） |
| priority | 'high' \| 'medium' \| 'low' | 文字列リテラル型。UIで表示しやすい |
| category | 'work' \| 'personal' \| 'shopping' | 文字列リテラル型。拡張しやすい |
| dueDate | string \| null | YYYY-MM-DD形式。nullは締切なしを表す |
| completed | boolean | true/falseで単純明快 |
| createdAt | string | ISO 8601形式。ソート・表示が容易 |

---

#### タスク配列の設計

```javascript
// すべてのタスクを格納する配列
const tasks = [
  {
    id: 1638360000000,
    title: "Phase 6の教材を完成させる",
    description: "Lesson 1-8とfinal-projectを作成",
    priority: "high",
    category: "work",
    dueDate: "2025-12-15",
    completed: false,
    createdAt: "2025-12-01T10:00:00.000Z"
  },
  // ...他のタスク
];
```

---

#### AIとの対話の進め方

仕様書作成フェーズでは、以下のようにAIと対話を重ねていくよ：

- データ構造の相談
- 画面構成の相談
- LocalStorage仕様の相談

#### 生成された仕様のチェックポイント

AIから仕様の提案をもらったら、以下をチェックしよう：

- **データ構造**
- **画面構成**
- **LocalStorage仕様**
- **実装可能性**

# Lesson 3

# UIデザインとプロトタイプ 🎨

---

## Lesson 3: UIデザインとプロトタイプ 🎨

**学習目標**：UIデザインの基本原則を理解し、AIと対話しながらUIデザインを作成し、静的プロトタイプ（JavaScriptなし）を作成できるようになる。

### なぜUIデザインとプロトタイプが重要か

仕様作成で「データ構造」と「機能仕様」が決まったね！でも、**「どんな見た目にするか」** はまだ決まっていないんだ。**UIデザインは、ユーザーが実際に触れる部分**。見た目が悪いと、機能が良くても使ってもらえないんだ！

```text
✅ 静的プロトタイプを先に作ると...

「まずHTMLとCSSで見た目だけ作ろう」
↓
カードのデザインが完成！
↓
「次はこのHTMLをJavaScriptで生成しよう」
↓
スムーズに実装できる！✨
```

**静的プロトタイプがあれば、実装時に迷わない**んだ！

---

## UIデザインとプロトタイピングのステップ

### ステップ1：UIデザインの基本を学ぶ

- 視認性（見やすさ）
- 操作性（使いやすさ）
- 一貫性（統一感）

### ステップ2：ワイヤーフレームを作成

- 画面のレイアウトを簡易的に描く
- テキストベースでOK（ASCIIアートやマークダウン）

---

### ステップ3：Tailwind CSSで静的HTMLを作成

- カードコンポーネントのデザイン
- レスポンシブレイアウト
- 色やフォントの選定

### ステップ4：プロトタイプを確認

- ブラウザで表示してみる
- モバイル・タブレット・デスクトップで確認
- 改善点をAIに相談

---

## 事例：カード型TODOアプリのUIデザインプロセス

### ステップ1：UIデザインの基本原則

#### 1. 視認性（Visibility）： **見やすく、情報が一目で分かる**

- 重要な情報（タイトル）は大きく
- 優先度は色で区別（高=赤、中=黄、低=緑）
- 完了したタスクは取り消し線＋半透明

#### 2. 操作性（Usability）： **直感的に操作できる**

- チェックボックスは左上（すぐに完了切替できる）
- 削除ボタンは右下（誤クリック防止）
- ボタンは大きめ（タッチしやすい）

#### 3. 一貫性（Consistency）： **デザインに統一感がある**

- カードの角丸は全て同じ（rounded-lg）
- ボタンのスタイルは統一
- 色使いは最小限（primary, secondary, accent）

---

### ステップ2：ワイヤーフレームの作成

**🤖 AIとの協働（最重要セクション）**

#### AIへの指示例

UIデザインフェーズでは、AIに「UIコンポーネントの生成」を依頼することが多いよ。

```text
「Tailwind CSSを使って、タスクカードのHTMLを作成してください。
【要素】
- タイトル（h3タグ、text-lg、font-bold）
- 説明（pタグ、text-gray-600、2行まで表示、超えたら省略記号）
- 優先度バッジ（bg-red-500/yellow-500/green-500、text-white、rounded-full）
- カテゴリ表示（bg-gray-200、text-gray-700、rounded-full）
- 完了チェックボックス（左上）
- 削除ボタン（右下、bg-red-500、rounded-lg）
【レイアウト】
- カード全体：bg-white、rounded-lg、shadow-md、p-6
- ホバー時：shadow-lg、少し浮き上がる
- 完了したタスク：opacity-60、タイトルに line-through
【レスポンシブ】
- モバイル：カード1カラム
- タブレット：カード2カラム
- デスクトップ：カード3カラム

静的HTML（JavaScriptなし）で、3つのサンプルカードを作ってください。」
```

# Lesson 4

# 機能の実装 ⚙️

---

## Lesson 4: 機能の実装 ⚙️

**学習目標**：AIに適切な指示を出し、基本機能から拡張機能まで段階的に実装できるようになる。

### なぜ段階的な実装が重要か

ここまでに、「何を作るか」（要件定義）、「どう作るか」（仕様書）、「どう見せるか」（UIデザイン）を決めたね！いよいよ**コードを書いて、動くものを作る**フェーズに入るよ！

---

### 段階的に実装する理由

いきなり全機能を実装しようとすると：❌ よくある失敗パターン

```text
「全部の機能を一気に実装しよう！」
↓
コードがぐちゃぐちゃに...
↓
どこでエラーが出ているか分からない...
↓
最初からやり直し 😭
```

でも、基本機能から段階的に実装すれば：✅ 成功するパターン

```text
「まずは追加と表示だけ作ろう！」
↓
動いた！✨
↓
「次は削除機能を追加しよう！」
↓
また動いた！✨
↓
少しずつ機能を追加して完成！🎉
```

**小さく作って、動作確認して、少しずつ拡張する**のが成功の秘訣！

---

## 機能の実装ステップ

### ステップ1：基本機能の実装

- 要件定義書、仕様書、UIデザインを入力に基本機能の実装をAIに依頼する

### ステップ2：機能の拡張

- 基本機能が動いたら、実用的なアプリに仕上げるための機能を追加する

### ステップ3：実用レベルに仕上げ

- 機能拡張版アプリに、データの永続化やUI改善を加えて、実用レベルに仕上げる

### ステップ4：最終調整とリファクタリング

- アプリのコードを整理し、プロフェッショナルなレベルに仕上げる

---

## 事例：カード型TODOアプリの基本実装

### ステップ1：基本機能の実装

まず、3つのファイルを作成するよ：

```text
card-todo/
  ├── index.html
  ├── styles.css
  └── script.js
```

---

**🤖 AIとの協働（最重要セクション）**

#### AIへの指示例：HTMLの作成

```text
「Tailwind CSSを使って、TODOアプリのHTMLを作成してください。

【要件】
1. Tailwind CSS 3.x をCDNで読み込む
2. 以下のセクションを含める：
   - ヘッダー（アプリタイトル「Card Todo」）
   - タスク追加フォーム（タイトル、説明、優先度、カテゴリ、締切日の入力欄）
   - 統計表示エリア（全タスク数、完了数、未完了数）
   - タスク一覧表示エリア（Gridレイアウト、id="task-list"）

【デザイン】
- ヘッダーは中央寄せ、大きなフォント、グラデーション背景
- フォームは最大幅800px、中央寄せ、影付きカード
```
