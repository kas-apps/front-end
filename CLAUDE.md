# プロジェクト固有の Claude 指示

このファイルは「バイブコーダーに贈るフロントエンド開発入門講座」プロジェクト専用の指示です。

---

## プロジェクト概要

### プロジェクトの目的

このプロジェクトは、**バイブコーディング（AI を活用したコーディングスタイル）を実践する人**に向けた、フロントエンド開発の入門講座を作成することを目的としています。

### ターゲット読者

- AI と協働してコードを書く新世代のエンジニア
- 「AI がコードを書いてくれるのに、なぜ基礎を学ぶ必要があるの？」という疑問を持つ人
- 効率的に AI を使いこなして、創造的な開発をしたい人

### プロジェクトの特徴

従来の教材とは異なり、以下の特徴を持っています：

- **AI との協働前提**：すべてを自分で書くのではなく、AI と一緒に開発する
- **理解と応用重視**：暗記ではなく、「なぜ」を理解する
- **実践的な組み合わせ**：技術単体ではなく、実際のプロジェクトで使える知識
- **バイブコーダー視点**：各レッスンに「AI への指示例」「コードの読み方」を含める

---

## 教材作成のガイドライン

### 文体・トーン

- **明朗快活**な口調で記述する
- 学習者に寄り添い、共感する表現を使う
- 「できる」「楽しい」「わくわくする」といったポジティブな言葉を積極的に使う
- 専門用語を使う際は、必ず平易な言葉で説明を添える

### 良い例・悪い例

```text
悪い例（堅苦しい、専門的すぎる）:
「CSS Grid Layoutはコンテナー要素に対してグリッドコンテキストを生成し、
子要素をグリッドアイテムとして配置する二次元レイアウトシステムである。」

良い例（分かりやすい、親しみやすい）:
「CSS Gridは、まるで方眼紙のようにページを分割して、要素を好きな場所に配置できる超便利な機能だよ！
縦にも横にも自由に配置できるから、複雑なレイアウトもラクラク！」
```

## 各レッスンの統一フォーマット

すべてのレッスンは以下の構成で統一してください：

### 1. レッスンタイトルと目標

レッスン名と学習目標を明記する。目標は 1-2 文で簡潔に。

### 2. 導入（Why）

- なぜこの技術を学ぶのか
- 実際にどんな場面で使うのか
- 学ぶことで何ができるようになるのか

### 3. 基礎知識（What）

- 技術の基本的な説明
- 重要な概念やキーワードの解説
- 図解や例え話を積極的に使う

### 4. コード例（How）

実際に動くコード例を提示：

- シンプルで理解しやすいコード
- コメントで各部分の役割を説明
- 実行結果も示す

### 5. バイブコーディング実践（最重要セクション！）

このプロジェクトの特徴的なセクション：

#### AI への指示例

「こう伝えると良いコードが生成される」という具体例を示す：

```text
良い指示の例:
「flexboxを使って、3つのカードを横並びに配置してください。
各カードは等幅で、間隔は20px、モバイルでは縦並びにしてください。」

曖昧な指示の例:
「カードをいい感じに並べて」
```

#### 生成されたコードの読み方

AI が生成したコードのどこを見るべきか、チェックポイントを示す：

- display: flex が設定されているか
- gap または margin で間隔が調整されているか
- メディアクエリでレスポンシブ対応されているか

#### よくある問題と修正方法

AI が生成したコードで起こりがちな問題と、その修正方法を説明。

例：

- 問題：カードの高さが揃わない
- 原因：align-items が設定されていない
- 修正：align-items: stretch を追加

### 6. 演習

演習の内容は教材とは別のファイルに作成し、教材からリンクする。

- 基礎編：簡単な練習問題（3-5 問）
- 応用編：少し難易度の高い課題（1-2 問）
- チャレンジ：創造性を発揮できる課題（任意）

### 7. まとめ

このレッスンで学んだことを箇条書きで整理し、次のステップへ誘導する。

## バイブコーダー向けの特別セクション

各レッスンには必ず以下の要素を含めてください：

### 1. AI への指示例

- 具体的で明確な指示の例を示す
- 良い指示と悪い指示を対比させる
- 技術用語を使った指示の効果を示す

### 2. コードリーディングのポイント

- 生成されたコードのどこを見るべきか
- 「このコードは良い/悪い」の判断基準
- 重要な部分とそうでない部分の見分け方

### 3. カスタマイズポイント

- 自分で微調整する際にいじる箇所
- 値を変えると何が変わるか
- よく調整する項目のリスト

### 4. よくあるトラブル

- AI が生成したコードで起こりがちな問題
- その原因と修正方法
- デバッグのヒント

## 技術的な方針

### コード例の書き方

1. **実際に動くコードを提示する**

   - コピペして試せるようにする
   - 必要な HTML と CSS の両方を含める
   - できるだけシンプルに保つ

2. **コメントを適切に入れる**

   - 重要な行には必ずコメントを付ける
   - 日本語のコメントを使う
   - 「なぜそうするか」を説明する

3. **段階的に複雑にする**
   - まず最小限のコードから始める
   - 徐々に機能を追加していく
   - 各ステップで何が変わったか明示する

### 良いコード例の提示方法

```html
<!-- 悪い例：コメントなし、いきなり複雑 -->
<div class="container">
  <div class="card">
    <img src="image.jpg" alt="Sample" />
    <h3>Title</h3>
    <p>Description</p>
    <button>Learn More</button>
  </div>
</div>

<!-- 良い例：段階的に説明 -->

<!-- ステップ1：まず基本的なカードを作る -->
<div class="card">
  <h3>カードのタイトル</h3>
  <p>カードの説明文</p>
</div>

<!-- ステップ2：画像を追加 -->
<div class="card">
  <img src="image.jpg" alt="カード画像" />
  <h3>カードのタイトル</h3>
  <p>カードの説明文</p>
</div>

<!-- ステップ3：ボタンを追加して完成 -->
<div class="card">
  <img src="image.jpg" alt="カード画像" />
  <h3>カードのタイトル</h3>
  <p>カードの説明文</p>
  <button>詳しく見る</button>
</div>
```

### ファイル・ディレクトリの命名規則

- **ディレクトリ名**：phase0-intro, phase1-html-css のように小文字とハイフン
- **レッスンディレクトリ**：01-html-basics, 02-css-basics のように番号付き
- **ファイル名**：README.md, index.html, styles.css など、分かりやすい名前
- **画像ファイル**：lesson01-example.png のようにレッスン名を含める

## プロジェクト構造の原則

### Phase 間の依存関係

- **Phase 0**: モチベーションと全体像の理解（必読）
- **Phase 1**: HTML/CSS 基礎（静的レイアウトのドキュメント作成、Phase 2 の前提）
- **Phase 2**: HTML/CSS 発展（Phase 1 の知識が必要、インタラクティブなレイアウト）
- **Phase 3**: JavaScript 基礎（Phase 1 の知識が必要、言語の基礎）
- **Phase 4**: JavaScript 発展（Phase 2, 3 の知識が必要、動的な機能実装）
- **Phase 5**: CSS フレームワーク（Phase 1-2 の知識が必要）
- **Phase 6**: 統合プロジェクト（Phase 1-5 の総まとめ）

### ディレクトリ構造の一貫性

各 Phase は以下の構造を守る：

```text
phaseXX-name/
├── README.md               # Phase全体の概要と学習ガイド
├── lessons/
│   ├── 01-topic/
│   │   ├── README.md       # レッスン教材
│   │   ├── exercises/      # 演習
│   │   │   ├── README.md   # 演習問題
│   │   │   └── solutions/  # 演習問題の解答例
│   │   │       ├── README.md # 解答例の解説
│   │   │       ├── 01-01.html
│   │   │       └── 01-02.html
│   │   └── resources/      # 教材で使用する画像など
│   └── 02-topic/
└── projects/               # 実践プロジェクト
    ├── project01/
    └── project02/
```

## 避けるべきこと

### 1. 専門用語の羅列

```text
悪い例：
「DOMツリーをトラバースして特定のノードを取得し、
イベントリスナーをアタッチする」

良い例：
「HTMLの要素を探して、クリックされた時に
何かする仕組みを作るよ！」
```

### 2. 理論だけの説明

```text
悪い例：
「Flexboxはフレキシブルボックスレイアウトモジュールの略称で...」

良い例：
「Flexboxを使うと、要素を横に並べたり中央に配置したりが
めちゃくちゃ簡単にできるよ！実際に試してみよう！」
```

### 3. バイブコーディング視点の欠如

すべてのレッスンに「AI とどう協働するか」の視点を必ず含める。
技術の説明だけで終わらせない。

### 4. 難易度の急激な上昇

前のレッスンで学んだことを前提に、少しずつ難易度を上げる。
いきなり複雑な例を出さない。

### 5. 実践から離れた内容

「実際のプロジェクトでは使わない」ような細かい仕様の説明は避ける。
実用的な知識を優先する。

## 教材更新時の注意点

### 新しいレッスンを追加する時

1. 既存のフォーマットに従う
2. Phase 内での位置づけを明確にする
3. 前提知識と学習目標を明記する
4. 「バイブコーディング実践」セクションを必ず含める

### 既存の教材を修正する時

1. 他のレッスンとの整合性を確認する
2. リンクが切れていないかチェックする
3. コード例が正しく動作するか確認する
4. 文体・トーンの一貫性を保つ

### 演習や課題を作成する時

1. 難易度を段階的に設定する（基礎 → 応用 → チャレンジ）
2. 解答例を必ず用意する
3. AI を使った解決方法も示す
4. 「この演習で何を学ぶか」を明記する

## デザイン・表現のガイドライン

### 絵文字の使用

プロジェクト全体で一貫した絵文字を使う：

- Phase 0（導入）
- Phase 1（HTML/CSS 基礎）
- Phase 2（HTML/CSS 発展）
- Phase 3（JavaScript 基礎）
- Phase 4（JavaScript 発展）
- Phase 5（フレームワーク）
- Phase 6（統合プロジェクト）
- AI に関する内容
- 演習・チャレンジ
- チェックリスト、まとめ
- コード例、実践
- ヒント、Tips

### 強調表現

- **太字**：重要なキーワード
- `コード`：技術用語、コマンド
- 引用：重要なポイントの強調
- リンク：参考資料、他のレッスンへの誘導

### 見出しレベル

```markdown
# レッスンタイトル（H1）

## 大セクション（H2）

### 中セクション（H3）

#### 小セクション（H4）
```

H4 より深い階層は使わない（読みづらくなるため）

## このプロジェクトのゴール

最終的に、このプロジェクトを通じて学習者が以下の状態になることを目指します：

1. **AI と効果的に協働できる**

   - 的確な指示が出せる
   - 生成されたコードを評価・修正できる

2. **フロントエンド開発の基礎を理解している**

   - HTML/CSS/JavaScript の基本が分かる
   - フレームワークを使いこなせる

3. **創造的な開発ができる**

   - 技術的な可能性を知っている
   - アイデアを形にできる

4. **自走できる**

   - わからないことを調べられる
   - 問題を自分で解決できる

5. **学習を楽しんでいる**
   - 「もっと学びたい」と思える
   - 成長を実感できる
